# Chapitre 1 — Comprendre Git : pourquoi et comment ?


## 1.1 Le problème que Git résout


Imaginez que vous travaillez sur un projet. Vous modifiez un fichier, puis vous réalisez que vous avez tout cassé. Sans outil adapté, vous n'avez aucun moyen de revenir en arrière proprement.


Beaucoup de gens adoptent alors cette mauvaise habitude :


```
projet_final.txt
projet_final_v2.txt
projet_final_VRAI.txt
projet_final_VRAI_definitif.txt
```


C'est ingérable, non ? C'est précisément ce problème que Git résout.


---


## 1.2 Qu'est-ce que Git ?


Git est un **système de contrôle de versions** (*Version Control System* — VCS). Il vous permet de :


- **Enregistrer** l'historique complet de vos modifications.
- **Revenir** à n'importe quel état antérieur de votre projet.
- **Travailler en parallèle** sur plusieurs versions d'un même projet.
- **Collaborer** avec d'autres personnes sans écraser leur travail.


> Git a été créé en **2005** par **Linus Torvalds** — le même homme qui a créé le noyau Linux — pour gérer le développement de ce noyau, qui impliquait des milliers de contributeurs.


---


## 1.3 Git vs GitHub — une distinction fondamentale


C'est une confusion très fréquente chez les débutants, clarifions-la dès maintenant :


| | **Git** | **GitHub** |
|---|---|---|
| **Nature** | Un logiciel | Une plateforme web |
| **Fonctionne** | Sur votre machine | Sur internet |
| **Rôle** | Gérer l'historique de vos fichiers | Héberger et partager vos projets Git |
| **Sans l'autre ?** | Oui, Git existe sans GitHub | Non, GitHub a besoin de Git |


Une analogie simple : **Git** est le moteur, **GitHub** est le garage où vous garez et montrez votre voiture.


---


## 1.4 Comment Git fonctionne-t-il concrètement ?


Git fonctionne comme une **série de photographies** de votre projet dans le temps. Chaque fois que vous le demandez explicitement, Git prend un **instantané** (*snapshot*) de l'état de vos fichiers. Cet instantané s'appelle un **commit**.


```
[Commit 1]──▶[Commit 2]──▶[Commit 3]──▶ ... (votre historique)
 "Début"      "Ajout       "Correction
               du menu"     du bug"
```


Vous pouvez à tout moment **voyager** dans cet historique et retrouver l'état exact de votre projet à un moment donné.

# Chapitre 2 — Configuration initiale de Git


Avant de créer votre premier projet Git, il est impératif de vous **présenter à Git**. En effet, chaque commit que vous créerez sera signé avec votre identité. Sans cette configuration, Git refusera de travailler proprement.


## 2.1 Les deux informations essentielles


Git a besoin de connaître :
- Votre **nom**
- Votre **adresse e-mail**


Ces informations seront attachées à chacun de vos commits, comme une signature.


## 2.2 La portée de la configuration


Git propose trois niveaux de configuration :


| Niveau | Portée | Option |
|---|---|---|
| **system** | Tous les utilisateurs de la machine | `--system` |
| **global** | Uniquement votre utilisateur | `--global` |
| **local** | Uniquement le projet en cours | `--local` |


Pour un usage personnel, nous utiliserons toujours `--global`.


## 2.3 Les commandes à exécuter


Dans votre Termux, saisissez les commandes suivantes en remplaçant par vos informations :


```bash
git config --global user.name "MrBot"
git config --global user.email "votre@email.com"
```


Puis configurons également l'éditeur de texte par défaut. Sous Termux, `nano` est le plus accessible :


```bash
git config --global core.editor nano
```


## 2.4 Vérification


Pour afficher l'ensemble de votre configuration Git :


```bash
git config --list
```

# Chapitre 3 — Votre premier dépôt Git


## 3.1 Qu'est-ce qu'un dépôt ?


Un **dépôt** (*repository*, souvent abrégé **repo**) est simplement un **dossier dont Git surveille le contenu**. Rien de plus. C'est le conteneur de votre projet.


Il en existe deux types :


| Type | Description |
|---|---|
| **Local** | Sur votre machine, dans Termux |
| **Distant** (*remote*) | Hébergé sur GitHub ou autre plateforme |


Pour l'instant, nous travaillons exclusivement en **local**.


## 3.2 Initialiser un dépôt


Commençons par créer un dossier de travail et l'initialiser comme dépôt Git :


```bash
mkdir mon_premier_repo
cd mon_premier_repo
git init
```


La commande `git init` crée un sous-dossier caché nommé `.git` à l'intérieur de votre dossier. C'est dans ce répertoire que Git stocke **tout** son historique et sa configuration locale. Il ne faut jamais le modifier manuellement.


Vous pouvez vérifier sa présence :


```bash
ls -la
```

## 3.3 Les trois zones de Git


Avant de créer votre premier commit, il est impératif de comprendre un concept central : **les trois zones dans lesquelles vos fichiers évoluent**.


```
┌─────────────────┐     git add      ┌─────────────────┐     git commit      ┌─────────────────┐
│  Working Tree   │ ───────────────▶ │  Staging Area   │ ──────────────────▶ │   Repository    │
│  (vos fichiers) │                  │   (l'index)     │                      │  (l'historique) │
└─────────────────┘                  └─────────────────┘                      └─────────────────┘
```


| Zone | Nom technique | Rôle |
|---|---|---|
| **Working Tree** | Arbre de travail | Vos fichiers tels que vous les éditez |
| **Staging Area** | Index | La salle d'attente — ce que vous préparez pour le prochain commit |
| **Repository** | Dépôt | L'historique permanent des commits |


### L'analogie du photographe


Imaginez que vous préparez une photo de groupe :


- Le **Working Tree** est la pièce où tout le monde se trouve.
- La **Staging Area** est l'endroit où vous **placez** les personnes que vous voulez sur la photo.
- Le **commit** est le **déclic** de l'appareil — la photo est prise et archivée définitivement.


> Cette zone intermédiaire qu'est la Staging Area est ce qui distingue Git de nombreux autres VCS. Elle vous donne un **contrôle précis** sur ce que vous incluez dans chaque commit, fichier par fichier, voire ligne par ligne.

## L'étape finale — `git commit`


Il est maintenant temps de prendre l'instantané. Un commit exige toujours un **message** qui décrit ce que vous avez fait. C'est une règle fondamentale.


### La convention d'un bon message de commit


Un message de commit doit être :
- **Court** — moins de 50 caractères pour le titre
- **Au présent et à l'impératif** — en anglais : `Add README`, pas `Added README`
- **Explicite** — un collègue doit comprendre ce qui a changé sans ouvrir le fichier


Exécutez votre premier commit :


```bash
git commit -m "Add README.md"
```


Partagez-moi le résultat. 🎓

 create mode 100644 README.md            MrBot $


Félicitations, MrBot ! 🎓


Vous venez d'effectuer votre **tout premier commit**. C'est un moment fondateur. Décryptons ensemble ce message de confirmation :

## Décryptage de la confirmation


```
[main (root-commit) f885655] Add README.md
```


| Élément | Signification |
|---|---|
| `main` | La branche sur laquelle le commit a été créé |
| `root-commit` | C'est le **tout premier commit** du dépôt — la racine de l'historique |
| `f885655` | L'**identifiant unique** du commit — appelé **hash SHA-1** |
| `Add README.md` | Votre message de commit |


```
1 file changed, 1 insertion(+)
```
> Un fichier modifié, une ligne ajoutée. Git comptabilise précisément chaque changement.


```
create mode 100644 README.md
```
> `100644` est le mode Unix du fichier — un fichier texte standard non exécutable.


---


## Le hash SHA-1 — une notion importante


Le `f885655` que vous voyez est en réalité une **empreinte unique de 40 caractères** dont Git n'affiche que les 7 premiers pour la lisibilité. Ce hash est **généré automatiquement** à partir du contenu du commit. Il est universel et unique — aucun autre commit au monde ne peut avoir le même identifiant.


> C'est ce hash qui vous permettra plus tard de **voyager dans le temps**, de **comparer des versions**, ou de **revenir en arrière**.

### Le concept de `HEAD`


`HEAD` est un pointeur qui répond à la question : **"Où suis-je en ce moment dans l'historique ?"**


```
[f885655] Add README.md
     ▲
   HEAD
   main
```


> Actuellement, `HEAD` et `main` pointent tous deux vers le même et unique commit. À mesure que vous créerez de nouveaux commits, `main` avancera automatiquement, et `HEAD` suivra.

# Les conventions de messages de commit


## La norme : Conventional Commits


Il existe une spécification largement adoptée dans l'industrie appelée **Conventional Commits**. Elle définit une structure standardisée :


```
<type>(<portée optionnelle>): <description courte>
```


---


## Les types fondamentaux


| Type | Usage | Exemple |
|---|---|---|
| `feat` | Ajout d'une **nouvelle fonctionnalité** | `feat: Add user login page` |
| `fix` | **Correction** d'un bug | `fix: Resolve null pointer on startup` |
| `docs` | Modification de la **documentation** | `docs: Update README installation steps` |
| `style` | **Mise en forme** sans changement de logique | `style: Fix indentation in main.py` |
| `refactor` | **Restructuration** du code sans ajout ni correction | `refactor: Simplify authentication logic` |
| `test` | Ajout ou modification de **tests** | `test: Add unit tests for login function` |
| `chore` | **Tâches de maintenance** sans impact sur le code | `chore: Update dependencies` |
| `perf` | **Optimisation** des performances | `perf: Reduce image loading time` |
| `ci` | Modifications liées à l'**intégration continue** | `ci: Add GitHub Actions workflow` |
| `build` | Modifications du **système de build** | `build: Update Dockerfile` |
| `revert` | **Annulation** d'un commit précédent | `revert: feat: Add user login page` |


---


## La portée — *scope* (optionnelle)


La portée précise **quelle partie** du projet est concernée :


```
feat(auth): Add password reset functionality
fix(api): Handle timeout errors gracefully
docs(readme): Add Termux installation guide
```

## Le concept de "detached HEAD"


En temps normal, voici la situation :


```
[f885655]──▶[788dbc0]──▶[2c29bf9]
```

Après votre `git checkout f885655`, la situation est :

## 4.4 Annuler des modifications — les trois scénarios


C'est une notion **critique**. En fonction de l'état de votre fichier, la commande d'annulation diffère :


| Scénario | Commande | Effet |
|---|---|---|
| Modification dans le Working Tree **non stagée** | `git restore <fichier>` | Annule les modifications — **irréversible** |
| Modification **stagée** | `git restore --staged <fichier>` | Retire du staging, conserve les modifications |
| Commit déjà effectué | `git revert <hash>` | Crée un **nouveau commit** d'annulation |


> ⚠️ `git restore` sans `--staged` est **destructif et irréversible**. Le fichier revient à l'état du dernier commit et vos modifications sont **définitivement perdues**.

## 4.5 `git revert` — annuler un commit proprement


### Le principe fondamental


`git revert` ne **supprime pas** un commit de l'historique. Il crée un **nouveau commit** qui fait l'inverse du commit ciblé.


```
Avant :
[f885655]──▶[788dbc0]──▶[2c29bf9]──▶[5729d04]
                                      "add line"
                                          ▲
                                        HEAD


Après git revert 5729d04 :
[f885655]──▶[788dbc0]──▶[2c29bf9]──▶[5729d04]──▶[nouveau commit]
                                      "add line"   "revert add line"
                                                         ▲
                                                       HEAD
```


> L'historique reste **intact et honnête**. C'est la méthode sûre et professionnelle — particulièrement importante lorsque vous travaillez en équipe.


### Pourquoi ne pas simplement supprimer le commit ?


> Supprimer un commit de l'historique (**git reset**) est une opération dangereuse en contexte collaboratif — elle réécrit l'historique et crée des conflits pour vos collaborateurs. Nous étudierons `git reset` plus tard avec toutes les précautions nécessaires.


---

| `git restore --staged` | Le staging | Staging Area | ✅ Oui | ❌ Non |
| `git revert <hash>` | Le commit entier | Repository | ✅ Oui | ✅ Oui |

Chapitre 5: Les branches
# Chapitre 5 — Les branches


## 5.1 Le problème sans les branches


Imaginez que vous travaillez sur un projet stable et fonctionnel. Vous souhaitez expérimenter une nouvelle fonctionnalité — mais si vous la développez directement sur votre projet principal et que ça ne fonctionne pas, vous risquez de tout casser.


Sans les branches, vous n'avez que deux options :
- Modifier directement le projet principal — **risqué**
- Copier tout le dossier — **archaïque**


Les branches de Git résolvent ce problème élégamment.


---


## 5.2 Qu'est-ce qu'une branche ?


Une branche est simplement un **pointeur mobile** vers un commit. Rien de plus. Elle vous permet de **diverger** du fil principal pour travailler en isolation.


```
                        main
                          ▼
[f885655]──▶[788dbc0]──▶[2c29bf9]
```


Lorsque vous créez une nouvelle branche, Git crée un **nouveau pointeur** vers le commit courant :


```
                        main
                          ▼
[f885655]──▶[788dbc0]──▶[2c29bf9]
                              ▲
                          feature
```


Les deux branches pointent vers le même commit — pour l'instant. Dès que vous committez sur `feature`, elles **divergent** :


```
                        main
                          ▼
[f885655]──▶[788dbc0]──▶[2c29bf9]──▶[nouveau]
                                         ▲
                                      feature
```


> `main` reste intact. Vous travaillez en totale isolation sur `feature`.


---


## 5.3 Les commandes essentielles


| Commande | Rôle |
|---|---|
| `git branch` | Lister les branches existantes |
| `git branch <nom>` | Créer une nouvelle branche |
| `git switch <nom>` | Se déplacer sur une branche |
| `git switch -c <nom>` | Créer **et** se déplacer en une commande |
| `git branch -d <nom>` | Supprimer une branche |


---

## La distinction à faire


Le préfixe de branche ne suit **pas exactement** la même logique que les types de commits. Il répond à la question :


> **"Quelle est la nature du travail effectué sur cette branche ?"**


---


## Les deux logiques comparées


| | Type de commit | Préfixe de branche |
|---|---|---|
| Documentation | `docs:` | `docs/` |
| Fonctionnalité | `feat:` | `feature/` |
| Correction | `fix:` | `fix/` |
| Refactoring | `refactor:` | `refactor/` |


---


## Donc dans notre cas précis


Nous ajoutons une **description** au README — c'est une modification de **documentation**. Vous avez donc raison :


```

❌  feature/add-description
```


> `feature/` est réservé aux branches qui ajoutent une **fonctionnalité au code** — une nouvelle page, un nouveau module, une nouvelle commande. Modifier un fichier de documentation ne constitue pas une fonctionnalité.

```
feature/   ──▶  nouvelle fonctionnalité dans le code
fix/       ──▶  correction de bug
docs/      ──▶  documentation uniquement
refactor/  ──▶  restructuration du code
test/      ──▶  ajout ou modification de tests
chore/     ──▶  maintenance, configuration
```

## Ce que Git a fait concrètement


```
                                              main
                                                ▼
[f885655]──▶[788dbc0]──▶[2c29bf9]──▶[5729d04]──▶[ed62fb1]
                                                       ▲
                                                  [501e838]
                                                       ▲
                                             docs/add-description
```


> En basculant sur `main`, Git a **physiquement modifié** les fichiers de votre Working Tree pour correspondre à l'état exact de `main`. Le commit `501e838` et ses modifications n'existent pas sur cette branche — ils sont isolés sur `docs/add-description`.

## La puissance que cela représente


| Situation | Sans branches | Avec branches |
|---|---|---|
| Tester une idée | Risque de casser le projet | Isolation totale |
| Travailler en parallèle | Impossible proprement | Naturel |
| Abandonner une expérimentation | Difficile | `git branch -d` |
| Intégrer le travail terminé | Copier-coller manuel | `git merge` |

## 5.6 Fusionner les branches — `git merge`


C'est l'étape naturelle après le développement sur une branche — **intégrer le travail dans `main`**.


```bash
git merge docs/add-description
```


> Vous êtes sur `main` — c'est la branche **réceptrice**. Git va y intégrer les commits de `docs/add-description`.


Exécutez puis partagez le résultat avec `git log --oneline` et `cat README.md`. 🎓

## Analyse du merge


### `git log --oneline`
```
501e838 (HEAD -> main, docs/add-description) docs: add description to README.md
ed62fb1 Revert "docs: add line to README.md"
...
```


> Les deux pointeurs `main` et `docs/add-description` sont maintenant sur le **même commit**. `main` a rattrapé `docs/add-description`.


### `cat README.md`
```

## Description
Dépôt d'apprentissage Git sous Termux.
```


> Le contenu de `docs/add-description` est maintenant intégré dans `main`.


---


## Ce qui s'est passé — Fast-forward


Git vous a effectué un merge de type **fast-forward** — le plus simple qui soit :


```
Avant le merge :


[ed62fb1]──▶[501e838]
    ▲              ▲
   main      docs/add-description




Après le merge :


[ed62fb1]──▶[501e838]
                  ▲
                main
                  ▲
          docs/add-description
```


> Un **fast-forward** se produit quand `main` n'a reçu **aucun commit** pendant que vous travailliez sur votre branche. Git se contente de **faire avancer le pointeur** de `main` vers le dernier commit de la branche — sans créer de commit de merge.


---


## Les deux types de merge


| Type | Condition | Résultat |
|---|---|---|
| **Fast-forward** | `main` n'a pas avancé pendant le travail sur la branche | Simple avancement du pointeur — pas de commit de merge |
| **Merge commit** | `main` a reçu des commits en parallèle | Création d'un commit de fusion qui réconcilie les deux historiques |


> Nous étudierons le **merge commit** et les **conflits** dans un prochain exercice.

## 5.7 Nettoyage — supprimer une branche fusionnée


Une fois une branche fusionnée, elle n'a plus de raison d'exister. La convention professionnelle est de la **supprimer** :


```bash
git branch -d docs/add-description
git branch
```


> `-d` pour **delete** — Git refusera de supprimer une branche **non fusionnée** avec ce flag, ce qui vous protège contre une suppression accidentelle.


Partagez-moi le résultat. 🎓


MrBot $ git branch
  docs/add-description
* main
MrBot $ git branch -d docs/add-description
Deleted branch docs/add-description (was 501e838).
MrBot $ git branch
* main
MrBot $


Parfait, MrBot ! 🎓


Propre et efficace. Notez le message de confirmation de Git :


```
Deleted branch docs/add-description (was 501e838)
```


> Git vous indique le dernier commit sur lequel pointait la branche avant suppression — une dernière traçabilité avant de l'effacer.

## 🔧 Les trois points à corriger


### 1 — `git branch -c` au lieu de `git switch -c`


```
git branch -c feat/add-gitignore   ❌
git switch -c feat/add-gitignore   ✅
```


> `-c` sur `git branch` signifie **copy** — il copie une branche existante. Vous avez eu de la chance que le résultat soit similaire, mais ce n'est pas la bonne commande. `git switch -c` crée **et** bascule en une seule commande.


### 2 — Le type de commit


```
feature: add .gitignore   ❌
chore: add .gitignore     ✅
```


> Un `.gitignore` est un fichier de **configuration et maintenance** du dépôt — pas une fonctionnalité. Le type approprié est `chore`. De plus, rappelez-vous que le type dans un commit est le nom court —  `feat` et non `feature`.


### 3 — Le préfixe de branche


```
feat/add-gitignore   ⚠️
chore/add-gitignore  ✅
```


> Par cohérence avec la nature du fichier, le préfixe devrait être `chore/` — comme le type de commit.

```
Branche    :  chore/add-gitignore
Commit     :  chore: add .gitignore
```
